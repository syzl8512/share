# reback (后悔药) 项目审计报告

## 1. 核心逻辑分析
`reback` 系统采用了 **"锁文件 + 后台守护进程"** 的经典看门狗架构：
- **触发端 (Skill)**：识别敏感操作，创建快照并埋下时间陷阱（锁文件）。
- **执行端 (Sentinel)**：后台静默计时，一旦超时则强制拉回状态。

## 2. 严格审核意见

### ✅ 优点
- **独立性强**：`Sentinel` 使用 LaunchAgent 独立于 Gateway 运行，即使 Gateway 崩溃或死循环，回滚依然生效。
- **配置简洁**：利用 `/tmp` 下的锁文件作为状态机，逻辑清晰。
- **快照化备份**：使用了带时间戳的目录 + 软链接模式，兼顾了历史追溯和主逻辑调用。

### ⚠️ 风险与不足

#### 1. 回滚原子性不足 (Atomicity)
- **风险**：`cp -rf "$BACKUP_DIR/"* "$CONFIG_DIR/"` 是递归复制。如果回滚过程中断（如断电），配置文件可能处于半新半旧的状态，导致系统彻底无法启动。
- **建议**：回滚应改用 `rsync --delete` 或者“先重命名旧目录，再复制新目录”的方式以保证一致性。

#### 2. 锁文件竞争风险 (Race Condition)
- **风险**：`openclaw-sentinel.sh` 在读取锁文件和执行回滚之间没有锁定保护。虽然在单用户环境下概率极低，但仍存在边缘冲突。
- **建议**：在 `Sentinel` 开始回滚前，先将锁文件 `mv` 到一个处理中文件名（如 `.lock.processing`），防止多次触发。

#### 3. 错误处理过于静默
- **风险**：`cp` 和 `rm` 操作使用了 `2>/dev/null`。如果磁盘空间不足或权限被篡改，回滚失败用户将无从知晓。
- **建议**：保留错误日志，并在 `sentinel.log` 中记录详细的失败栈。

#### 4. 确认逻辑的单向性
- **风险**：如果用户回复的“确认”消息由于网络波动或网关卡顿没被 Skill 处理，`Sentinel` 仍会触发回滚。
- **建议**：确认成功后，Skill 应向用户回显“✅ 保护已解除”。

## 3. 建议修改方案

### 建议修改 `openclaw-sentinel.sh` (原子性增强版)
```bash
# ... 原有变量定义 ...

# 改进的回滚核心逻辑
rollback() {
    log "⚠️ 开始原子回滚..."
    # 1. 验证备份完整性
    if [ ! -f "$BACKUP_DIR/openclaw.json" ]; then
        log "❌ 错误: 核心备份文件缺失，取消回滚"
        return 1
    fi

    # 2. 原子替换配置 (使用 rsync 保持一致性)
    # 如果没有 rsync，建议重命名原目录进行备份
    tar -czf "$CONFIG_DIR/pre_rollback_state.tar.gz" -C "$CONFIG_DIR" . 2>/dev/null
    
    # 执行恢复
    cp -rf "$BACKUP_DIR/"* "$CONFIG_DIR/"
    
    # 3. 重启服务
    launchctl kickstart -k ai.openclaw.gateway 2>/dev/null
    log "✅ 原子回滚与服务重启指令已发送"
}
```

### 建议修改 `SKILL.md` (安全增强)
- **增加清理逻辑**：建议在备份前先清理 30 天前的旧快照，防止 `~/.openclaw/backups` 撑爆磁盘。
- **环境检查**：在创建锁文件前，检查 `launchctl list | grep ai.openclaw.gateway`。如果服务本身就没启动，保护逻辑应跳过。

## 4. 总结
这是一个设计精巧的“后悔药”。目前的实现已经能解决 90% 的误操作场景。建议采纳 **原子性复制** 和 **回滚预备份** 的策略，将其从“好用”提升到“工业级可靠”。
