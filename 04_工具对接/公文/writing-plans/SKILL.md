# Writing Plans - 任务规划

## 概述

当你有需求或设计文档，需要拆分为可执行任务时使用。

**开场声明**："我正在使用 writing-plans 技能来创建实施计划。"

**任务粒度**：每个任务 2-5 分钟可完成

## 计划文档头部

每个计划**必须**以这个头部开始：

```markdown
# [功能名称] 实施计划

> **注意：** 使用 workflow-runner 技能按任务逐个执行。

**目标：** 一句话描述要构建什么

**架构：** 2-3 句话说明方法

**技术栈：** 关键技术/库

---
```

## 任务结构

```markdown
### 任务 N: [组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

**步骤 1: [具体行动]**

描述要做什么，包含完整代码或命令。

**步骤 2: [验证步骤]**

运行什么命令，预期什么结果。

**步骤 3: [提交]**

添加具体提交信息。
```

## 任务拆分原则

每个任务应该是：
- **可独立执行** - 不依赖其他任务的结果
- **可验证** - 有明确的成功标准
- **2-5分钟** - 不要拆得太细

### 好任务示例

- "安装依赖包" ✅
- "配置数据库连接" ✅
- "创建用户表模型" ✅

### 坏任务示例

- "实现用户功能" ❌ (太大)
- "做一些配置" ❌ (不明确)

## 输出格式

计划完成后，展示给用户并提供执行选项：

**"计划已保存到 `docs/plans/<filename>.md`。两个执行选项：**

**1. 串行执行** - 按顺序执行每个任务，验证后再继续下一个

**2. 并行执行** - 使用 workflow-runner 并行执行独立任务

**3. 子代理执行** - 使用 workflow-runner 的子代理模式

**你想用哪种方式？"**

## 与 Superpowers 对比

| Superpowers | OpenClaw |
|-------------|----------|
| 开发者执行 | 子代理执行 |
| TDD 测试 | 可选的验证步骤 |
| Git 提交 | 灵活的提交策略 |

## 核心原则

- **精确文件路径** - 始终使用精确路径
- **完整代码** - 不要"添加验证"，要完整代码
- **精确命令** - 包含预期输出
- **DRY** - 不要重复
- **YAGNI** - 不要过度设计
